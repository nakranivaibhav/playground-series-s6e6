# kaggleforge — cross-competition memory (case bank)

Lessons that generalize **across** competitions. RETRIEVE the relevant ones
*before* proposing a node (retrieve-before-propose), and RETAIN a new one *after*
any promotion or hard-won failure. One line per lesson, ruthlessly short.
Per-competition state lives in `comps/<slug>/`, never here.

Format: `- [<tag>] <lesson> — <why> (<slug | general>)`

## Validation & leakage
- [cv] Trust a well-built CV over the public LB; a CV↔LB gap is a diagnostic, not a demote. — chasing the public LB causes private shake-up (general)
- [leak] Target-encode/scale/impute INSIDE the fold only. — fitting across the fold leaks the val target (general)
- [leak] If an id/entity repeats across rows, GroupKFold is mandatory. — random folds put the same group in train+val (general)
- [leak] Time column ⇒ TimeSeriesSplit + past-only features (no centered windows, no global stats). — future info leaks backward (general)
- [leak] Leakage gating is STATIC code scan only (`tools/leakage_scan.py`: fit-in-fold / target-in-features / id-order / group / temporal / duplicates). The per-node shuffled-label control was removed — it retrained the model on permuted labels and doubled compute on slow nodes (NN/DART) for a check the static scan covers. (general)
- [seed] Freeze folds once; one seed controls ONLY the split. — a seed that also drives init/data order leaks across re-runs (general)

## Search & modelling
- [tree] One atomic change per node so every CV delta is attributable. — bundled changes hide which one helped (general)
- [family] Keep ≥2 model families alive; try a second family before tuning the first. — diversity beats early hyperparameter tuning (general)
- [pivot] After ~5 stalled improves, pivot the architecture, not the hyperparameters. — long flat lineages signal a wrong frame (general)
- [tabular] LightGBM/XGBoost/CatBoost are the default tabular workhorses. — strong, fast, robust to mixed types (general)
- [ensemble] Only blend arms whose per-fold failures are DISJOINT. — correlated arms add no signal and cost a slot (general)
- [ensemble] Blend by fold-honest weighted probability-AVERAGE (weights picked on other folds, scored on held-out fold) for a robust baseline. BUT for balanced-accuracy/imbalanced multiclass the top-of-LB move is a STACK: multinomial LogisticRegression(class_weight='balanced') on the base OOF LOG-probs + a per-class threshold search (differential_evolution over 3 multipliers) maximizing the metric, ALL under honest nested CV. The class_weight='balanced' is essential — a plain multinomial meta inherits majority bias and regresses below its bases. (playground-series-s6e6 discussions; Chris Deotte GM) (general)
- [noise] On a noisy metric near the ceiling the public LB is resolution-limited (top teams within < private 1σ): CV can rise while LB falls. Monte-Carlo over private-sized slices shows "pick 2 decorrelated submissions" adds ~0 expected value — lock your 2 highest-CV blends and stop chasing decorrelation/4th-decimal LB. (playground-series-s6e6) (general)
- [fe] Verify whether categorical features are just deterministic cuts of existing numerics before engineering on them — target-encoding a redundant categorical adds noise (n18 regressed; spectral_type/galaxy_population were exact color-index thresholds). (playground-series-s6e6)
- [ensemble] A balanced-LogReg STACK lifts ONLY from genuinely-new STRONG families; everything else fails two ways: (a) WEAK bases hurt it even when de-correlated — a plain MLP (0.966), LogReg (0.939), ExtraTrees (0.961) each dragged a 0.9695 stack by −0.0001 to −0.0002 (strength matters as much as de-correlation); (b) a 2nd SEED or 2nd CONFIG of an ALREADY-INCLUDED family washes/hurts — 3 RealMLP seeds added +0.00003 over 1, a wider-RealMLP config −0.00012, a Lossguide-CatBoost −0.00003 — too correlated with the family member already in. What DID lift the stack each by +0.0001..+0.0003: the FIRST member of each new strong family (RealMLP→+0.0026 breakthrough, then TabM, LightGBM, CatBoost on the same rich FE). So: stack one strong member per family, stop adding within-family variants, and the stack saturates once all families are covered. (playground-series-s6e6) (general)
- [ensemble] Blends SATURATE on diversity: once ONE strong de-correlated NN arm (TabM) is in, a 2nd NN family (FT-Transformer) and a DART tree both get 0 weight in the fold-honest search — more arms of an already-covered flavor add nothing. Add a NEW direction, not another of the same. — n14/n15/n16 all failed to beat the 4-arm champion (playground-series-s6e6)
- [dart] LightGBM `boosting_type='dart'` is ~O(trees²) and ignores early-stopping (it rebuilds/rescales the whole ensemble each round). 600 deep trees × multiclass was >1h/fold; trim to ~250 trees / shallow leaves or skip it — and DART rarely earns blend weight anyway. (playground-series-s6e6)
- [ensemble] A balanced-LogReg STACK saturates the same way a blend does: a base must be strong AND de-correlated. NN bases that are diverse but WEAK (RealMLP 0.950, TabPFN-3 0.943 vs the trees' 0.962–0.965) REGRESS the honest stacked OOF (−0.0002 each) instead of lifting it; even a properly-retrained CatBoost (+0.0014 solo) added only +0.00006 (noise) to a saturated GBDT stack. The discussion-ceiling's +0.003 needs NN bases IN the 0.963–0.968 band — so the lever to close a stack's gap-to-ceiling is NN-base STRENGTH (e.g. TabPFN context too thin at 10k of 577k; RealMLP on default config), not more families. (playground-series-s6e6)
- [ensemble] A diversity arm helps a blend only if it's ALSO strong solo — low error-correlation alone is not enough. — an MLP at err-corr 0.72 vs GBDTs but ~0.010 lower solo CV earned only 0.05 weight and didn't beat the GBDT-only blend (playground-series-s6e6)
- [ensemble] A STRONGER arm can HURT a blend if it's more correlated: a model retrained on the same features as an existing arm makes the same errors. — XGBoost on the full feature set beat its base-feature self solo (+0.0005) but swapping it into the blend lost -0.00013, because it shared the LightGBM arm's features (err-corr 0.88 vs 0.85); the blend valued the base-feature arm's diversity more (playground-series-s6e6)
- [fe] Feature-parity across arms is NOT always good for a blend — deliberately giving arms DIFFERENT feature sets is a cheap diversity source. — close a feature gap for the single-model score, but keep a feature-diverse arm alive for the ensemble (general)
- [fe] An IMPORTANT feature does NOT mean re-encodings of it help — a strong tree already extracts a raw feature; adding sin/cos/cartesian/grid-cell versions usually just overfits. — positional re-encodings of the dominant `delta` regressed LightGBM −0.001/6σ leak-free; raw delta+galactic coords already captured it (playground-series-s6e6)
- [fe] High-cardinality native categoricals (e.g. a 400+ grid-cell id) are a prime GBDT overfit source — they carve fold-specific noise. — prefer low-card bins, target-encoding with strong smoothing fit-inside-fold, or just the continuous coords (general)
- [reference] drop-column importance ≠ split/gain importance: gain ranks REPLACEABLE-but-used features high (redundant colors), drop-column ranks IRREPLACEABLE features high (unique signal like sky position). Read both. (general)
- [nn] On clean tabular features a plain MLP sits ~0.01 below tuned GBDTs; expect it to lose solo and rarely rescue a blend. — try GBDTs first; only invest in an NN (TabM/tuned) if you NEED a strong de-correlated arm (general)
- [nn] A weak NN result is usually an UNDER-BUILT model, NOT the model's ceiling — verify you built the PROVEN recipe before concluding it caps. On one comp our RealMLP scored 0.949 (bare pytabkit-TD on bare features) and we wrongly declared "NNs cap ~0.95, we're at the dataset ceiling". A user-flagged public top-LB notebook revealed its single RealMLP scored 0.96973; porting that proven recipe — heavy FE (color pairs, redshift ratios, integer-floor cat views, fit-in-fold TargetEncoder) + a fully-specified RealMLP (n_ens internal ensemble, [512]³, PBLD periodic embeddings, ScalingLayer, expm4t dropout, EMA) — got us 0.969 SOLO (+0.020), which alone beat our whole stack and lifted the champion 0.9666→0.9692 (LB 0.96722→0.96993). The bare library default (pytabkit RealMLP_TD) genuinely under-performs; the public recipe hand-rolls for a reason (rule-8 "library critically fails" exception). LESSON: pull a public reference notebook EARLY (kaggle kernels pull / KaggleApi.kernels_pull) and diff your base against it; a 0.02 solo gap is a build bug, not a ceiling. (playground-series-s6e6) (general)
- [nn] BUT genuinely de-correlated-yet-weak NN/foundation bases (TabPFN v2/2.5/v3 0.94-0.95, TabICL 0.959, FT-T 0.957) do NOT lift a balanced-LogReg stack once a STRONG NN (TabM, or a properly-built RealMLP) already fills the de-correlated-NN slot — each adds −0.0001 to 0. Strength matters as much as de-correlation; don't bet many rounds on weak-but-diverse bases. (playground-series-s6e6) (playground-series-s6e6)
- [perf] In-context/foundation models (TabPFN/TabICL): `predict()` re-runs the WHOLE stored context every call and caches nothing between calls — so predict the entire query block in ONE call (or large chunks), never a small per-batch loop that re-encodes a big context each time. A 4k-row loop over a 50k context (worsened by an OOM handler flooring the batch at 512 → ~225 re-encodes) made a run 4 h that is 10 min when the context is encoded once (peak VRAM also dropped 28→5 GB). ALWAYS time one fold/unit and project the total before launching a multi-hour job. (playground-series-s6e6) (general)
- [nn] TabM (lib `tabm` + `rtdl_num_embeddings`, k=32 ensemble + PiecewiseLinear embeddings) reaches GBDT-level solo AND stays de-correlated (err-corr ~0.82 vs ~0.86 among trees) — it's THE tabular NN that lifts a GBDT blend where a plain MLP can't. — TabM earned the largest weight (0.35) and added +0.00036 over the GBDT-only blend (playground-series-s6e6)
- [nn] For a brand-new GPU (e.g. RTX 5090 / Blackwell sm_120) install torch from the matching CUDA index (`uv add torch --index https://download.pytorch.org/whl/cu128`) and verify a real GPU matmul, not just torch.cuda.is_available(). — stock wheels lack the new sm_ kernels (general)
- [ablate] A/B every added block against its parent CV; reject what regresses. — feature/tool additions sometimes hurt (general)
- [external] Before betting on external/extra-source training data, run a 2-min PRE-CHECK: (1) adversarial AUC of origin (own-train vs external) on shared features — ~0.5 safe, >0.8 risky, >0.95 abort; (2) per-feature KS to see WHICH columns drift. Then prove it with a single-base honest A/B (external rows added to the TRAIN side of each frozen fold ONLY, val stays pure own-data). On synthetic-from-real Playground comps the real source data DRIFTS (the generator reshapes distributions): SDSS17 concat gave adversarial AUC 0.909 (redshift KS 0.194 worst) and HURT a LightGBM −0.0010, and adversarial-propensity downweighting made it WORSE (−0.0012), not better — drift this strong isn't fixable by reweighting. Don't rebuild a whole base zoo on external data before this cheap A/B clears. (playground-series-s6e6) (general)
- [stack] Once a balanced-LogReg stack is family-saturated, the META config is NOT the lever: logit vs raw-log-prob features, C∈{1,0.3,0.1}, and seed-bagging the meta all washed within 0.12·sem on a 15-base set. A gap to a higher public stack is then base-set difference (more/stronger bases), not stacker tuning — stop tuning the meta. (playground-series-s6e6) (general)
- [external] On a synthetic-from-real Playground comp, the "original-data PRIOR features" trick that the whole top-LB cluster shares (P(class|color-bin-key) computed on the original dataset, mapped onto train/test — NOT row-concat) can be INERT for an already-rich-FE stack: it washed both as a standalone GBDT base (stack −0.00012) and as an in-place upgrade of our strongest RealMLP arm (solo flat, stack −0.0002). Two reasons: (1) rich engineered colour FE already encodes that structure, (2) the priors carry the generator's drift (same shift that makes row-concat hurt). So a top public NOTEBOOK's recipe ported faithfully can still fail to lift YOUR stack — the value may be irreducibly tied to their fold scheme + base alignment, not transferable feature tricks. (playground-series-s6e6) (general)
- [stack] A stack base that is ITSELF a label-fit sub-model hand-targeted at a known error pocket is a CV-MIRAGE hazard — gate it on the LB, not CV. A LightGBM "GALAXY-vs-STAR specialist" trained only on the low-redshift confusion zone, added as a 16th base, lifted nested-CV +0.001073 (~3.4 sem) and passed the static leak scan CLEAN — but its public LB CRASHED −0.0080 (0.97043→0.96242). Mechanism: the specialist double-uses the SAME train labels the meta also fits, and its narrow-zone OOF is systematically more optimistic on train than its full-train model is on test; that optimism sits in BOTH the OOF and the nested holdout (they share the specialist's train-fit bias), so honest nested CV can't see it. Ordinary single-level stacking double-uses labels safely; a narrow hand-targeted specialist amplifies it into an invisible overfit. Don't promote such a base on CV alone. (playground-series-s6e6) (general)
- [cv] A CV↔LB gap that is ~10× the public-slice noise (e.g. −0.0086 when noise ~0.00087) is NOT noise and overrides "trust CV": it signals the CV gain is an artifact — revert. The "trust CV over LB" rule is for NOISE-level gaps; a gap an order of magnitude beyond noise is a genuine generalization-failure signal. When a single node both (a) beats CV by a suspiciously clean margin and (b) tanks the LB, believe the LB and roll back. (playground-series-s6e6) (general)
- [stack] Multi-seed FOLD RE-PARTITION of the meta (re-run StratifiedKFold under K seeds, average the stacked OOF+test) is the structural element that separates many public top stackers from a single-partition stack — and it is DISTINCT from the dead "seed-bag the meta weights" lever (which averages fits on the SAME folds → same expected weights → wash). BUT it only lifts if meta-fit variance is your binding constraint: on a saturated, already-strong base set it was NEUTRAL (+0.00004, holdout-confirmed honest). The decisive value is diagnostic — when multi-seed meta averaging does nothing, your gap to a higher public stack is the BASE SET (more/stronger/de-correlated/fold-aligned bases), NOT the stacker mechanism. Stop tuning the meta loop after this check. (playground-series-s6e6) (general)
- [stack] When the meta is maxed (multi-seed re-partition gives 0) AND every public model FAMILY is already a base, do a base-set DIFF against the public top to find a genuinely MISSING architecture — for us the only one absent was a DCN/CrossNet (explicit multiplicative feature-cross inductive bias; distinct from residual-MLP/tree/attention). Worth building once. But note even THAT washed (standalone 0.966 < TabM, re-stack −0.00001): on a saturated, color-feature-driven tabular comp the cross-interaction bias added no de-correlated signal the trees+TabM didn't already span. After the missing-architecture check also washes, the residual gap to a higher public stack is fold-scheme ALIGNMENT (their whole zoo shares one split; faithful ports to your folds cap ~0.0013 low) + private noise — NOT a missing model. Stop hunting bases. (playground-series-s6e6) (general)
- [nn] Feature->image->CNN (DeepInsight / GAF-recurrence "spectrogram" / color-color images) is a WASH on low-dimensional tabular data — do not spend many rounds on it. The audio-spectrogram analogy FAILS because a waveform is genuinely high-dimensional (thousands of correlated samples with a temporal manifold a CNN uncovers), whereas a tabular row is a few near-independent scalars (here ~6: redshift+4 colors+brightness) with NO within-row manifold — any synthesized image is an invertible re-encoding of features the trees already own, so the CNN re-learns the same boundaries and its errors correlate with the existing stack (node_0056 1D-CNN err-corr 0.78; node_0057 image-ResNet capped BA 0.940 << TabM 0.968). Even a physically-motivated NON-circular transform (a (1+z) rest-frame wavelength warp that re-measures colors on a redshift-stretched axis the stack never forms) ADDED NOISE — warp-OFF scored higher than warp-ON in a fold-0 ablation. Gate any such attempt with a CHEAP fold-0 kill-switch (standalone BA floor + err-corr-vs-stack > 0.6 -> kill before the full run) so it costs ~20min to falsify, not a full 5-fold. ADDENDUM (node_0058): a pretrained ImageNet backbone (YOLO26n-cls + Muon/MuSGD, 2 epochs) is WORSE than a from-scratch ResNet on these synthetic GAF/RP images (0.888 vs 0.940) — natural-image inductive bias is counterproductive on tiny non-photographic encodings; and an EXTENSIVE augmentation tournament (photometric input-jitter / mixup-cutout-erasing / flip-rotate-shear-crop) moved BA by only ±0.003 — augmentation cannot rescue an information-starved encoding. The ceiling is the encoding's information content, not the trainer or the augmentation. (playground-series-s6e6) (general)
- [repro] Faithfully porting a top public base to YOUR frozen folds caps BELOW its published score because the score is fold-scheme-specific (their StratifiedKFold seed ≠ yours): a 0.969 reference xgb reproduced at 0.9677 on our folds. Don't chase a public single-model number you can't reproduce without adopting their exact folds — and adopting them means rebuilding your whole aligned base zoo. (playground-series-s6e6) (general)

## Process
- [budget] CV decides WHAT to submit; never spend a slot to A/B on the LB. — 5/day is scarce, the LB is the OOD check not the selector (general)
- [metrics] Never promote on CV-mean alone; one outlier fold can carry the mean. — inspect per-fold deltas (general)
- [403] A 403 on download/submit means rules-not-accepted/unverified, NOT bad creds. — the #1 misdiagnosis (general)
